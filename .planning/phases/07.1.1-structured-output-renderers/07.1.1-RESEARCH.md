## Summary

Phase 07.1.1 replaces the stub JSON summary card in `page.tsx` (the `isValidJSON` branch) with five typed React renderer components — one per `ChannelType`. The input is always a parsed `StructuredOutput` (typed in `src/types/index.ts`); the output is visual previews that mimic how end-users would experience each channel. All AI, scoring, and streaming logic stays untouched.

The existing codebase already has excellent precedents to build on: `output-card.tsx` shows the exact card shell, badge, save/copy/adjust action row, horizontal scroll pattern, and framer-motion entrance animation. The new renderers must conform to those same patterns while adding the visual fidelity of real email clients, SMS bubbles, iOS notification cards, and wireframe screen cards.

The main design challenge is faithfully representing real-world UI conventions using only Tailwind CSS and shadcn/UI components — no new packages needed. Each renderer can achieve its target visual (email client strip, iMessage bubble, iOS lock-screen notification, mobile wireframe card) through careful use of border-radius, background colour, spacing, and typography from the existing palette.

**Primary recommendation:** Build a single `StructuredOutputRenderer` entry component that accepts a `StructuredOutput` and routes each `Channel` to its typed sub-renderer. Multi-tier grouping logic lives at this orchestration layer. Sub-renderers are pure presentational components that accept a single channel interface object as their only prop.

### Pattern 1: StructuredOutputRenderer (Orchestrator)

**What:** Top-level component that receives a `StructuredOutput`, groups `Channel[]` by `type+tier`, and renders sub-renderers in sequence with the shared action row.

**When to use:** Called from `page.tsx` in the `isValidJSON` branch to replace the stub summary card.

```typescript
// src/app/(dashboard)/chat/structured-output-renderer.tsx
"use client";

import { motion } from "framer-motion";
import { StructuredOutput, Channel, TierLabel } from "@/types";
import { EmailRenderer } from "./renderers/email-renderer";
import { SMSRenderer } from "./renderers/sms-renderer";
import { PushRenderer } from "./renderers/push-renderer";
import { UXJourneyRenderer } from "./renderers/ux-journey-renderer";
import { AdCopyRenderer } from "./renderers/ad-copy-renderer";
import { OutputCardActions } from "./output-card"; // reuse existing action row

interface Props {
  output: StructuredOutput;
  messageId: string;
  onSave: () => void;
  onAdjust: (channelIndex: number) => void;
}

export function StructuredOutputRenderer({ output, messageId, onSave, onAdjust }: Props) {
  // Group channels: same type + non-null tier → multi-tier group
  // Different type channels → rendered sequentially
  const groups = groupChannels(output.channels);

  return (
    <div className="space-y-3">
      {groups.map((group, i) => (
        <motion.div
          key={`${messageId}-group-${i}`}
          initial={{ opacity: 0, y: 4 }}
          animate={{ opacity: 1, y: 0 }}
          transition={{ duration: 0.25, ease: "easeOut", delay: i * 0.06 }}
        >
          <ChannelGroupRenderer group={group} onSave={onSave} onAdjust={() => onAdjust(i)} />
        </motion.div>
      ))}
    </div>
  );
}

// Multi-tier group = same channel type, multiple tier variants
type ChannelGroup = { channels: Channel[] };
function groupChannels(channels: Channel[]): ChannelGroup[] { ... }
```

**Key decision:** `groupChannels` treats same-`type` channels with non-null `tier` as a single group (rendered with Tabs). Different `type` channels always get separate cards.

**What:** When `output.channels` contains the same `type` with multiple different non-null `tier` values (e.g., Classic + VIP email), group them into a `Tabs` component.

**When to use:** Detected by the `groupChannels()` function in `StructuredOutputRenderer`.

### Anti-Patterns to Avoid

- **Separate action row components per renderer:** Reuse `OutputCardActions` from `output-card.tsx`. Don't copy-paste.
- **Dynamic Tailwind class concatenation:** Never build class strings like `"bg-" + color`. Write complete class strings. Tailwind PurgeCSS strips dynamic classes (existing codebase comment confirms this).
- **Parsing typed data as markdown:** The new renderers receive `Channel` interfaces with typed fields. There is no markdown parsing needed — access fields directly.
- **Streaming renderer activation:** Never activate structured renderers while `isCurrentlyStreaming` is true. The gate in `page.tsx` (`isValidJSON && !isCurrentlyStreaming`) already handles this — don't duplicate or break it.
- **New `message-flash` animation inside renderers:** The `animate-message-flash` class is applied at the message-div level in `page.tsx`, not inside the renderer. Don't re-implement it.

## Common Pitfalls

### Pitfall 1: Importing from "motion/react" vs "framer-motion"

**What goes wrong:** The `framer-motion` package at v12 introduced a `motion` package as a rebrand, but the project's `package.json` installs `framer-motion` (not `motion`), and all existing files import from `"framer-motion"`. Importing from `"motion/react"` would cause a module-not-found error.

**Why it happens:** The Framer/Motion rebranding is confusing — they're different package names at this point.

**How to avoid:** Always import `{ motion, AnimatePresence }` from `"framer-motion"` (same as output-card.tsx, score-card.tsx, page.tsx).

**Warning signs:** TypeScript error "Cannot find module 'motion/react'."

---

### Pitfall 2: Tailwind Dynamic Class Purging

**What goes wrong:** Building class strings dynamically (e.g., `"rounded-" + size`) causes those classes to not be included in the production build, since Tailwind's content scanner can't detect them statically.

**Why it happens:** Tailwind PurgeCSS scans source files for complete class strings. Dynamically assembled strings aren't detected.

**How to avoid:** Write complete class strings in code. Use `cn()` with static class strings and conditional values (e.g., `cn("bg-indigo-600", isActive && "ring-2")`).

**Warning signs:** Classes work in dev but disappear in production builds.

---

### Pitfall 3: Activating Renderers During Streaming

**What goes wrong:** If `isValidJSON()` returns true on a partial JSON stream, the renderer will attempt to parse incomplete JSON and crash or show broken UI.

**Why it happens:** The Output Agent streams the JSON character-by-character. Early in streaming, the opening `{` makes `isValidJSON()` return false (good), but mid-stream it might pass the `JSON.parse` without throwing.

**How to avoid:** The existing gate in `page.tsx` is:
```typescript
} : isValidJSON(message.content) ? (
```
This is only reached when `!isCurrentlyStreaming`. Never move `isValidJSON` logic before the streaming check. The `StructuredOutputRenderer` should be called only from this already-gated branch.

**Warning signs:** Renderer appears mid-stream showing partial data.

---

### Pitfall 4: UX Journey Step Width Overflow

**What goes wrong:** If step cards are given a percentage width instead of a fixed `flex-shrink-0` width, they collapse when the container is wider than expected.

**Why it happens:** Inside `overflow-x-auto`, flex items need `flex-shrink-0` and an explicit width to maintain their size.

**How to avoid:** Use `w-[360px] flex-shrink-0` on each step card. The existing `JourneyScreenCard` uses `w-[400px] flex-shrink-0` — match this or use `w-[360px]` for the slightly richer typed renderer.

**Warning signs:** Step cards collapse to minimum content width in horizontal scroll.

---

### Pitfall 5: Missing Null Guards on Optional Fields

**What goes wrong:** `UXJourneyStep.heading`, `cta_label`, `helper_text`, `error_text` are all `string | null`. `PushChannel.deep_link_label` is `string | null`. Rendering without null guard causes "null" to appear as text.

**Why it happens:** TypeScript allows assigning null to string | null, but JSX renders null as empty — however `{field}` where `field = null` renders nothing, which is fine; the issue is `{field && <Component />}` works correctly only with the `&&` guard.

**How to avoid:** Use `{field && <div>{field}</div>}` for every optional field. Never use `{field!}` non-null assertions in JSX.

**Warning signs:** The word "null" appears in rendered UI.

---

### Pitfall 6: OutputCardActions Copy Button Content

**What goes wrong:** The Copy button in `OutputCardActions` copies its `body` string prop. For structured renderers, this prop should be the human-readable text content of the channel, not the raw JSON.

**Why it happens:** The existing stub just copies `message.content` (the full JSON string). The new renderers need to construct a meaningful text representation for each channel.

**How to avoid:** Each renderer should compute a `copyText` string from the channel fields and pass it to `OutputCardActions` as the `body` prop. For email: subject + body + CTA. For SMS: message. For push: title + body. For UX journey: each step as text. For ad copy: headline + body + CTA.

**Warning signs:** Clicking Copy pastes raw JSON.

---

### Pitfall 7: Tier Badge on Multi-Tier Tabs

**What goes wrong:** When rendering a multi-tier Tabs group, showing a tier badge in the card header alongside the tab triggers is redundant and visually noisy.

**How to avoid:** When a channel group has multiple tiers (rendering as Tabs), omit the tier badge from the card header — the tab trigger labels (Classic, Plus, VIP) serve that purpose. Only show tier badge when a single-tier channel renders as a standalone card.

## Open Questions

1. **`copyText` format for each renderer**
   - What we know: `OutputCardActions` takes a `body: string` prop for Copy button. For JSON outputs, copying the raw JSON is useless.
   - What's unclear: Should the copy text be formatted plaintext, markdown, or a human-readable summary?
   - Recommendation: For each channel, construct a simple plaintext string: email = "Subject: ...\n\nHeadline: ...\nBody: ...\nCTA: ..."; SMS = message only; Push = "Title: ...\nBody: ..."; etc. This is what a copywriter would actually paste into a doc.

2. **`onAdjust` mapping from structured output to `AdjustTarget`**
   - What we know: `AdjustTarget.sectionBody` is what gets sent as context to the AI. It's currently the raw markdown section body.
   - What's unclear: Should the `sectionBody` for a structured channel be the raw JSON or a human-readable representation?
   - Recommendation: Pass the serialised channel object as JSON string for `sectionBody`. The AI prompt construction in `submitAdjust()` (page.tsx) already truncates to 200 chars anyway.

3. **Should tier-null channels (no tier) render differently?**
   - What we know: `channel.tier` is `TierLabel | null`. Most single-channel responses will have `tier: null`.
   - What's unclear: Should a `tier: null` card show no tier badge, or some "All tiers" label?
   - Recommendation: Show no tier badge when `tier === null`. Only show tier badge when a specific TierLabel is set. This matches current `output-card.tsx` pattern where tier is shown as context only.
