# Phase 07.1.1: Structured Output Renderers - Context

**Gathered:** 2026-02-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the current JSON summary card placeholder in the chat interface with rich, type-specific renderers for each output channel. Each channel type should render as a styled preview that represents what the final end-user UI would look like — using shadcn/UI components to visualise headers, subheaders, body copy, buttons, input fields, and other UI elements as they would appear to the brand's customers.

The input is the parsed `StructuredOutput` JSON (already typed in `src/types/index.ts`). The output is a set of React components that render each channel's content visually in the chat. All data fetching, AI streaming, scoring, and Figma logic stay untouched.

</domain>

<decisions>
## Implementation Decisions

### Overall Approach
- Each channel type gets its own renderer component that visually represents the final customer-facing UI
- Rendered inside the chat as part of the message thread — replacing the current JSON summary card
- Built with shadcn/UI components: Card, Badge, Button, Input, Separator, etc.
- Each rendered output retains Save, Copy, and Adjust buttons below it

### Channel Renderers Required

**Email**
- Preview that looks like an email client render (not a code view)
- Subject line styled as a large bold heading
- Preheader as smaller muted subtitle
- Sections rendered in order:
  - hero: prominent headline + optional subheadline
  - body: readable prose copy
  - cta: styled button representation (shadcn Button) with optional supporting text
  - footer: small muted text

**SMS**
- Phone message bubble style — single bubble on the right (as if from the brand) or left-aligned in a message thread
- Shows the full message text in a rounded bubble
- Character count displayed below (e.g. "61 / 160 chars")

**Push Notification**
- Styled to look like a real iOS/Android notification card
- App icon placeholder, app name, timestamp placeholder
- Bold title, body text below it
- Optional deep link label as a small secondary action

**UX Journey**
- Each step rendered as a phone/screen frame or clean card
- Step number + screen name as header
- Heading (if present) as screen title
- Body copy as screen body text
- CTA as a styled button
- Helper text as small muted guidance copy below button
- Error text as red warning copy (if present)
- Steps displayed horizontally scrollable (as established in Phase 7.1) but with richer internal content

**Ad Copy**
- Clean ad preview card
- Headline bold and prominent
- Body copy below
- CTA as a styled button

### Multi-Channel + Multi-Tier Grouping
- When a response contains multiple channels, each renders as its own card in sequence
- When a response contains the same channel for multiple tiers (Classic/Plus/VIP), they render side by side or in a tab/toggle view — Claude's discretion on layout
- Tier badge displayed on each card

### Streaming Behaviour
- During streaming: keep current MemoizedMarkdown plain rendering (no change)
- After streaming completes and content is valid JSON: switch to the typed renderer
- This is already gated by `isValidJSON()` and `isCurrentlyStreaming` in page.tsx

### Claude's Discretion
- Exact colour of SMS bubble (should match indigo/zinc palette)
- Push notification mock app icon and name
- Whether multi-tier same-channel cards use tabs, side-by-side, or stacked layout
- Animation on channel card entrance (align with existing message-appear pattern)
- Whether to show a "Preview" label or output type header on each card
- Exact phone frame style for UX Journey steps (can be a simple rounded rectangle)

</decisions>

<specifics>
## Specific Ideas

- The email renderer should feel like you're previewing an actual email — not reading copy in a box
- The SMS bubble should feel like iMessage or WhatsApp — a real bubble, not a card with "SMS" written on it
- Push notifications should look like the iOS lock screen notification format — recognisable at a glance
- UX Journey steps should feel like low-fidelity wireframe screens — structured layout showing where elements sit, not just text
- The goal is for someone to look at the rendered output and immediately understand what it would look like in the final product

</specifics>

<deferred>
## Deferred Ideas

- Dark mode variants for each renderer
- Interactive state simulation (hover states on CTA buttons)
- Device frame chrome (actual iPhone/Android device outline) — too heavy for this phase
- Export to Figma integration

</deferred>
