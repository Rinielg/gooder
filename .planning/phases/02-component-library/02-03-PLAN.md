<objective>
Create three reusable skeleton loading variants (Card, Table, Form) that can replace spinner-based loading states across all pages.

Purpose: Skeleton loading provides a better UX than spinners by showing the shape of upcoming content, reducing perceived load time. These three variants cover the three primary content patterns in Gooder: card lists (profiles, outputs), data tables (standards, objectives, definitions), and forms (settings, profile editing). Page-specific compositions can be built from these in later phases.

Output: A single file with three exported skeleton components, all using pulse fade animation.
</objective>

    **Skeleton design decisions (LOCKED):**
    - Pulse fade animation (shadcn default)
    - Content-shaped: skeletons mirror the layout they replace (circle for avatar, short bar for title, longer bars for text)
    - Three variants: Card, Table, Form

    **CardSkeleton:**

    Props:
    ```typescript
    interface CardSkeletonProps {
      count?: number;     // Number of card skeletons to render (default: 3)
      className?: string;
    }
    ```

    Layout (mirrors a typical card in the app -- e.g., profile cards, output cards):
    - Wrapped in the `Card` component from `@/components/ui/card`
    - Inside the card:
      - Top row: circle skeleton (40x40, `rounded-full`) for avatar + short bar (w-32, h-4) for title + shorter bar (w-20, h-3) for subtitle
      - Middle: 3 lines of text skeletons with varying widths (100%, 80%, 60%) and `h-3` height
      - Bottom row: short bar (w-24, h-8, rounded-md) for action button
    - Card uses `p-6` padding, `space-y-4` vertical spacing
    - Renders `count` cards in a `grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-4` layout

    **TableSkeleton:**

    Props:
    ```typescript
    interface TableSkeletonProps {
      rows?: number;      // Number of data rows (default: 5)
      columns?: number;   // Number of columns (default: 4)
      className?: string;
    }
    ```

    Layout (mirrors data tables -- e.g., standards, objectives, definitions pages):
    - Full-width container with `rounded-lg border`
    - Header row: `columns` number of skeleton bars at `h-4` with varying widths (first column wider: w-40, rest: w-24) on a `bg-muted/50` background with `px-4 py-3`
    - Data rows: `rows` rows, each with `columns` cells of skeleton bars at `h-3` with varying widths per column (randomize slightly: 75%, 50%, 60%, 40%). Use `px-4 py-3` per row with `border-t`
    - The first data column in each row should be wider (w-48) than the others (w-20 to w-32)

    **FormSkeleton:**

    Props:
    ```typescript
    interface FormSkeletonProps {
      fields?: number;    // Number of form fields (default: 4)
      className?: string;
    }
    ```

    Layout (mirrors form layouts -- e.g., settings, profile editing):
    - Vertical stack with `space-y-6`
    - Each field:
      - Label skeleton: `h-3 w-20` (small bar above input)
      - Input skeleton: `h-10 w-full rounded-md` (full-width rectangle matching input height)
    - After all fields: button skeleton at bottom: `h-10 w-32 rounded-md` (submit button shape)
    - Optional: first field can be wider label (`w-32`) to mimic "Email" vs "Name" label width variation

    **All skeletons share:**
    - Use the base `Skeleton` component: `<Skeleton className="..." />`
    - The base Skeleton already applies `animate-pulse bg-muted rounded-md` from shadcn
    - Each exported skeleton accepts a `className` prop for additional styling via `cn()`

    **Discretion (skeleton show delay):** Do NOT add a built-in delay mechanism. Let consumers control this via conditional rendering or Suspense boundaries. Keeping skeletons stateless makes them composable.

    Export all three: `{ CardSkeleton, TableSkeleton, FormSkeleton }`
