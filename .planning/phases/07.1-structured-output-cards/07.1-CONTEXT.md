# Phase 7.1: Structured Output Cards - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Restructure how AI responses are rendered in the chat when they contain multiple output types (UX journeys, SMS, push notifications, emails, WhatsApp, etc.). Each output type becomes its own separate visual card in the chat with its own Save, Copy, and Adjust button. A focused Adjust dialog above the input lets users reference a specific output for targeted revisions. All streaming, scoring, and Figma logic stays untouched.

</domain>

<decisions>
## Implementation Decisions

### Card Container Style
- Clean branded cards — design-system-native, no device chrome/mockups
- Each output type has a coloured pill badge:
  - SMS → purple
  - Email → blue
  - Push Notification → orange
  - WhatsApp → green
  - UX Journey → indigo
  - Other/generic → zinc/neutral
- Each output type renders as its own separate message card (not grouped in one AI message)
- Each card has its own Save button, Copy button, and Adjust button below it

### UX Journey Layout
- Horizontal scrollable flow — screens laid out left-to-right in a scrollable row
- Each screen card: 380–420px wide (fits 2–3 visible at once)
- Sequential numbered/titled cards per screen/step/platform
- Arrows or visual connectors between screens optional (Claude's discretion)

### Message-Type Field Rendering (Email, Push, SMS, WhatsApp)
- **Email**: Subject line (visually distinct at top) + Body copy + CTA button label — all with character counts
- **Push Notification**: Title + Subtitle + Body — with character counts per field
- **SMS**: Message text with 160-character count indicator
- **WhatsApp**: Message text with character count
- Character limits to enforce visually: Email subject ~60, Push title ~65, Push subtitle ~45, SMS 160, WhatsApp 1024
- Fields detected by parsing patterns in the AI output (e.g. "Subject:", "Title:", "Body:", "CTA:")

### Adjust Flow
- Each output card has an "Adjust" button
- Clicking Adjust opens a focused mini-dialog ABOVE the input area (not a modal, not a full page)
- Dialog shows: the output content + a text field "What changes do you want?"
- Submitting sends a targeted message referencing that specific output type and content
- Dialog closes after sending

### Output Type Detection
- AI responses split at `###` headings using existing `splitIntoSections()` logic
- Section title used to detect output type:
  - UX Journey / Journey / User Flow / Onboarding Flow → UX Journey card
  - Email → Email card
  - SMS / Text Message → SMS card
  - Push / Push Notification → Push card
  - WhatsApp → WhatsApp card
  - Anything else with a title → Generic branded card
- If only one section (no `###` headings) → render as plain MemoizedMarkdown (no change)
- Intro/preamble text before the first `###` renders as a plain intro block above the cards

### Claude's Discretion
- Exact colour tokens for each badge type (should use existing palette: indigo, zinc, green, etc.)
- Visual connector between UX journey screen cards (arrow icon or dashed line)
- Exact character count styling (progress bar vs number only)
- Adjust dialog exact dimensions and positioning
- Animation on card entrance (should align with existing message-appear pattern)
- Handling of output types that don't match any known type

</decisions>

<specifics>
## Specific Ideas

- The horizontal scroll flow for UX journeys should feel like a user flow — reading left to right naturally
- Cards should look like they came from the same design system as the rest of the app — not feel like embedded widgets
- The Adjust dialog should be unobtrusive — sits just above the input area and feels native to the chat
- Character counts should feel informative not alarming — a subtle count below each field, turning red/amber only when over limit

</specifics>

<deferred>
## Deferred Ideas

- Realistic device mockups (iPhone frame for SMS, email client preview) — user chose clean cards instead
- Tabbed UX journey layout — user chose horizontal scroll
- Inline editing of output text — user chose Adjust dialog pattern

</deferred>
