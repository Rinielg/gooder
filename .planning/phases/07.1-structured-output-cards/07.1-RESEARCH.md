## Summary

This phase adds a new rendering layer between the AI response (`message.content`) and the chat UI. When an assistant message contains multiple `###`-headed sections after streaming completes, each section is rendered as a distinct visual card instead of one combined MemoizedMarkdown block. The `splitIntoSections()` function in `page.tsx` already performs the split — the work is in building the card components, wiring the detection gate, and threading the Adjust dialog state.

The critical architectural constraint is that the `messages[]` array is **not modified**. Each AI message remains a single entry in the array. The splitting and visual separation happens purely at render time — one logical message fans out to N visual cards. This avoids any impact on scoring, streaming, or Figma logic.

The Adjust dialog is a floating panel positioned above the input form, not a modal. Its state (`adjustTarget: { messageId, sectionTitle, sectionBody } | null`) lives in `page.tsx` alongside other chat state. Because it sits outside the message list render loop, toggling it does not re-render message cards.

**Primary recommendation:** Extract an `OutputCardGroup` component that receives `sections: ContentSection[]` and `messageId`, renders the appropriate card variant per section, and feeds Adjust state up to `page.tsx` via callback. Keep all new components in the chat directory as separate files to avoid further bloating `page.tsx`.

### Pattern 1: Streaming Gate — Only Activate After `status === "ready"`

**What:** Output card detection and rendering is conditional on streaming being complete. During streaming, render `MemoizedMarkdown` as before. When `status === "ready"` and message has multiple sections, switch to `OutputCardGroup`.

**Why:** `splitIntoSections()` on a partially-streamed response will produce garbage partial sections. The `###` heading marker may not have arrived yet. The last section body is always incomplete mid-stream.

**How to implement:**

```typescript
// In the message render loop in page.tsx
const sections = useMemo(
  () => splitIntoSections(message.content),
  [message.content]
);
const isMultiSection = sections.filter(s => s.title).length >= 2;
const isStreaming = message.id === streamingMessageId; // derive from status + message position

// During streaming OR single-section: plain rendering
if (isStreaming || !isMultiSection) {
  return <MemoizedMarkdown content={message.content} id={message.id} />;
}

// After streaming completes and multiple sections exist:
return (
  <OutputCardGroup
    sections={sections}
    messageId={message.id}
    onSave={saveOutput}
    onAdjust={openAdjust}
  />
);
```

**Streaming message detection:** The currently streaming message is the last assistant message when `status === "streaming"`. Use `isStreaming = status === "streaming" && message.id === lastAssistantId` where `lastAssistantId` is computed from `displayMessages`.

### Pattern 2: Virtual Cards from One Message — No messages[] Mutation

**What:** A single `messages[]` entry fans out to multiple visual cards at render time. The message object itself is unchanged.

**Structure:**

```typescript
// OutputCardGroup maps sections to cards
function OutputCardGroup({ sections, messageId, onSave, onAdjust }) {
  const titled = sections.filter(s => s.title);
  const intro = sections.find(s => !s.title);

  return (
    <div className="space-y-3">
      {intro && (
        <div className="text-sm text-muted-foreground">
          <MemoizedMarkdown content={intro.body} id={`${messageId}-intro`} />
        </div>
      )}
      {titled.map((section, i) => {
        const type = detectOutputType(section.title);
        return (
          <OutputCard
            key={`${messageId}-card-${i}`}
            section={section}
            outputType={type}
            messageId={messageId}
            sectionIndex={i}
            onSave={onSave}
            onAdjust={onAdjust}
          />
        );
      })}
    </div>
  );
}
```

### Pattern 3: Output Type Detection — Regex on Section Title

**What:** Pure function, no state, derives output type from section title string.

```typescript
type DetectedOutputType =
  | "email" | "sms" | "push" | "whatsapp" | "ux_journey" | "generic";

function detectOutputType(title: string): DetectedOutputType {
  const t = title.toLowerCase();
  if (/ux journey|user journey|user flow|onboarding flow|journey/i.test(t)) return "ux_journey";
  if (/\bemail\b/i.test(t)) return "email";
  if (/\bsms\b|text message/i.test(t)) return "sms";
  if (/push notification|push copy|\bpush\b/i.test(t)) return "push";
  if (/whatsapp/i.test(t)) return "whatsapp";
  return "generic";
}
```

**Confidence:** HIGH — this is a pure string matching function; the AI output format uses these exact terms per the system prompt.

### Pattern 4: Structured Field Parsing — Regex Within Section Body

**What:** For email/push/SMS/WhatsApp cards, parse the section body for labelled fields. Handles variations like "Subject:", "**Subject:**", "Subject Line:".

```typescript
interface ParsedFields {
  subject?: string;    // email
  title?: string;      // push
  subtitle?: string;   // push
  body?: string;       // email, push, sms, whatsapp
  cta?: string;        // email
  message?: string;    // sms, whatsapp fallback
}

function parseStructuredFields(body: string): ParsedFields {
  const fields: ParsedFields = {};

  // Strip markdown bold markers for matching
  const clean = body.replace(/\*\*/g, "");

  const extract = (pattern: RegExp): string | undefined => {
    const m = clean.match(pattern);
    return m?.[1]?.trim() || undefined;
  };

  fields.subject = extract(/^subject(?:\s+line)?:\s*(.+)$/im);
  fields.title = extract(/^title:\s*(.+)$/im);
  fields.subtitle = extract(/^subtitle:\s*(.+)$/im);
  fields.body = extract(/^body(?:\s+copy)?:\s*([\s\S]+?)(?=\n(?:cta|subject|title|subtitle):|$)/im);
  fields.cta = extract(/^cta(?:\s+button)?(?:\s+label)?:\s*(.+)$/im);
  fields.message = extract(/^message:\s*([\s\S]+?)$/im);

  return fields;
}
```

**Note:** If no structured fields are found (no `Subject:` etc. labels), fall back to rendering the entire body as `MemoizedMarkdown`. This handles AI responses that don't use the labelled format.

**UX Journey screen parsing:** The AI outputs UX journey screens as nested content under the journey section. Screens are typically separated by `##` or `####` subheadings, or by numbered `**Screen 1:**` / `**Step 1:**` bold headers.

```typescript
function parseUxJourneyScreens(body: string): Array<{ title: string; content: string }> {
  // Try splitting on #### or ## subheadings first
  const byHeading = body.split(/^#{2,4}\s+/m).filter(Boolean);
  if (byHeading.length > 1) {
    return byHeading.map(part => {
      const nl = part.indexOf("\n");
      return nl === -1
        ? { title: part.trim(), content: "" }
        : { title: part.slice(0, nl).trim(), content: part.slice(nl + 1).trim() };
    });
  }

  // Try splitting on bold numbered headers: **Screen 1:** or **Step 1:**
  const byBold = body.split(/\n\*\*(?:Screen|Step|Platform|State)\s+\d+[:.]\*\*/i);
  if (byBold.length > 1) {
    // Reconstruct with titles from the split pattern
    const titles = [...body.matchAll(/\*\*(?:Screen|Step|Platform|State)\s+\d+[:.]\*\*/gi)].map(m => m[0]);
    return byBold.slice(1).map((content, i) => ({
      title: titles[i]?.replace(/\*\*/g, "").trim() ?? `Screen ${i + 1}`,
      content: content.trim(),
    }));
  }

  // Fallback: whole body as one screen
  return [{ title: section.title, content: body }];
}
```

### Pattern 8: Adjust Dialog — State in page.tsx, Rendered Near Input

**What:** The Adjust dialog is NOT a modal. It is a `motion.div` positioned above the input form. Its state lives in `page.tsx` to prevent unnecessary re-renders of the message list.

```typescript
// In page.tsx — new state
interface AdjustTarget {
  messageId: string;
  sectionTitle: string;
  sectionBody: string;
  outputType: DetectedOutputType;
}
const [adjustTarget, setAdjustTarget] = useState<AdjustTarget | null>(null);
const [adjustText, setAdjustText] = useState("");

function openAdjust(target: AdjustTarget) {
  setAdjustTarget(target);
  setAdjustText("");
}

function submitAdjust() {
  if (!adjustTarget || !adjustText.trim()) return;
  const prompt = `For the ${adjustTarget.sectionTitle}: ${adjustText.trim()}\n\nOriginal content:\n${adjustTarget.sectionBody}`;
  sendMessage({ text: prompt });
  setAdjustTarget(null);
  setAdjustText("");
}
```

**Positioning:** Render the Adjust dialog between the messages area and the input form, using `AnimatePresence` + `motion.div`:

```typescript
// Between the scroll container and the input form in page.tsx JSX
<AnimatePresence>
```

### Anti-Patterns to Avoid

- **Mutating messages[]:** Never add, modify, or split entries in the AI SDK messages array. All card splitting is presentation-only at render time.
- **Running splitIntoSections() on streaming content:** Gate behind `status === "ready"` or by checking that the message is not the currently-streaming message. Partial `###` headers mid-stream break section detection.
- **Putting adjustTarget state inside OutputCard:** Adjust state belongs in page.tsx. If it were inside each card, closing the dialog would require prop drilling back up, and multiple cards could conflict.
- **Using ScrollArea for horizontal UX Journey scroll:** The existing `ScrollArea` component only adds a vertical scrollbar indicator. For horizontal scroll, use native `overflow-x-auto` with the existing global webkit scrollbar CSS. It already styles `height: 6px` scrollbars.
- **Extending SectionedMessage:** `SectionedMessage` in `markdown-message.tsx` was prototyped but reverted. Do NOT build on it. Create new `OutputCardGroup`/`OutputCard` components.
- **Re-rendering message list when Adjust dialog opens:** Adjust state must not be stored inside the message map loop or any component that is a child of the message list.

## Common Pitfalls

### Pitfall 1: Streaming Section Detection Firing Prematurely
**What goes wrong:** `splitIntoSections()` is called on partial content during streaming. A message body of `"Here's your content:\n\n### Email\n\nSubject: Welcome to "` produces a section with incomplete body. The card renders mid-thought.
**Why it happens:** The section render path is not gated on streaming status.
**How to avoid:** Compute `isStreamingThisMessage` for each message. Only switch from `MemoizedMarkdown` to `OutputCardGroup` when that flag is false.
**Warning signs:** Cards appearing and disappearing during streaming; character counts jumping.

### Pitfall 2: Section Body Contains Unparsed Markdown Labels
**What goes wrong:** The AI writes `**Subject:** Welcome email` with bold markers. The field parser `^Subject:\s*(.+)` doesn't match because `**` is prepended.
**Why it happens:** AI frequently bolds field labels.
**How to avoid:** Strip `**` before running field regexes: `body.replace(/\*\*/g, "")`.
**Warning signs:** StructuredFieldCard renders its fallback (full MemoizedMarkdown) instead of individual fields.

### Pitfall 3: WhatsApp Save Fails DB Constraint
**What goes wrong:** `saveSection()` passes `type: "whatsapp"` to Supabase insert. The DB has `CHECK (type IN ('ux_journey', 'email', 'sms', 'push'))`. Insert fails silently or throws.
**Why it happens:** `OutputType` in `types/index.ts` doesn't include `"whatsapp"`. The DB constraint also doesn't include it.
**How to avoid:** Map `"whatsapp"` → `"sms"` for DB save (closest semantic match). Also add `"whatsapp"` to `OutputType` in `types/index.ts` if saving as distinct type is important, AND add a DB migration. Given phase scope, the safe choice is the mapping.
**Warning signs:** `toast.error("Failed to save output")` when saving WhatsApp cards; Supabase `new row violates check constraint` error in console.

### Pitfall 4: UX Journey Body Has No Subheadings — One Blank Screen
**What goes wrong:** The AI outputs a UX journey without `####` or `**Screen:**` markers. `parseUxJourneyScreens()` returns one item — the whole body — and renders one large card instead of a scroll of screens.
**Why it happens:** AI output format is not guaranteed; the system prompt may not enforce exact screen separators.
**How to avoid:** The fallback (one card with full MemoizedMarkdown body) is acceptable — it is not broken, just not visually split. Document this as expected behaviour. The Adjust flow still works.
**Warning signs:** UX Journey card is unexpectedly wide with a single very tall inner card.

### Pitfall 5: Adjust Dialog Causes Message List Re-render on Every Keystroke
**What goes wrong:** `adjustText` state is in a parent component that renders the message list. Every keystroke in the Adjust textarea triggers re-render of all messages, causing visible lag.
**Why it happens:** State placed at the wrong level — too high if it triggers expensive subtrees.
**How to avoid:** Extract `AdjustDialog` as its own component that manages its own `value`/`onChange` internally. Only call `onSubmit(text)` when submitted. The parent's `adjustTarget` (open/close) state is cheap; the per-keystroke `adjustText` stays inside `AdjustDialog`.
**Warning signs:** Keyboard input in Adjust textarea feels sluggish; React DevTools shows message list children re-rendering on keypress.

### Pitfall 6: Badge Colour Classes Purged by Tailwind
**What goes wrong:** Dynamic badge class strings like `"bg-" + colour + "-100"` are not detected by Tailwind's content scanner. Classes are purged from the production build.
**Why it happens:** Tailwind only includes classes that appear as complete strings in scanned files.
**How to avoid:** Use the complete class strings in the `OUTPUT_TYPE_CONFIG` object (e.g. `"bg-purple-100 text-purple-800"`). Never build class names dynamically from string parts.
**Warning signs:** Badge colours work in dev but appear unstyled in production build.

### Pitfall 7: Adjust Prompt Leaks Entire Section Body Verbatim Into Chat History
**What goes wrong:** The targeted Adjust message visible to the user in chat is cluttered with the full original output repeated as quote.
**Why it happens:** Naive prompt construction includes the full `sectionBody` in the user-visible message.
**How to avoid:** The targeted message text sent to `sendMessage()` should be concise. Pass the original content as context but frame it: `"Please adjust the ${sectionTitle}: ${adjustText}. Reference the previous ${outputType} output."` — let the AI refer to its own prior message rather than echoing the whole body. Or include a truncated excerpt (first 200 chars) rather than the full body.
**Warning signs:** Chat history shows very long user messages that repeat the AI output.

```typescript
// Source: page.tsx line 607-610

function OutputCard({ section, outputType, messageId, sectionIndex, onSave, onAdjust }) {
  const config = OUTPUT_TYPE_CONFIG[outputType];

  return (
    <motion.div
      initial={{ opacity: 0, y: 4 }}
      animate={{ opacity: 1, y: 0 }}
      transition={{ duration: 0.25, ease: "easeOut", delay: sectionIndex * 0.05 }}
      className="rounded-xl border border-border bg-white shadow-elevation-1"
```

## Critical Findings (Must-Know for Planning)

### Finding 1: WhatsApp is NOT a Valid DB OutputType
**Source:** `src/types/index.ts` line 181 + `supabase/migration.sql` line 83

The TypeScript type `OutputType = "ux_journey" | "email" | "sms" | "push"` does not include `"whatsapp"`. The database has a hard CHECK constraint: `CHECK (type IN ('ux_journey', 'email', 'sms', 'push'))`. Inserting `type: "whatsapp"` will fail.

**Decision for phase:** Map `"whatsapp"` → `"sms"` when saving to DB. This avoids a migration and keeps scope tight. WhatsApp cards still render as WhatsApp (with green badge), they just save as type `"sms"` in the library. Add a comment in code noting this mapping.

### Finding 2: ScrollArea Component is Vertical-Only
**Source:** `src/components/ui/scroll-area.tsx` — only mounts a vertical `ScrollAreaScrollbar`

The existing `ScrollArea` component does not support horizontal scrolling without modification. For UX Journey horizontal scroll, use a plain `<div className="overflow-x-auto">`. The global CSS in `globals.css` already styles webkit scrollbars to 6px height — this applies to horizontal scrollbars too.

### Finding 3: Badge Component Accepts className Override
**Source:** `src/components/ui/badge.tsx` — uses `cn(badgeVariants[variant], className)`

The Badge component merges `className` after the variant styles, allowing complete colour override. Pass the full colour class string directly. No new variants needed.

### Finding 4: framer-motion v12 Is Installed
**Source:** `package.json` `"framer-motion": "^12.34.0"`

v12 introduced motion APIs. The codebase already uses `motion`, `AnimatePresence` from page.tsx imports. Use the same import pattern: `import { motion, AnimatePresence } from "framer-motion"`.

### Finding 5: page.tsx Is ~900 Lines — Extract Components
**Source:** Direct file read — 899 lines

All new components MUST be in separate files. The `score-card.tsx` pattern is the model to follow: component in its own file, receives callbacks as props, no direct state imports from page.tsx.

### Finding 6: saveOutput() Currently Uses Whole-Message Type Detection
**Source:** `page.tsx` lines 498-503

The current `saveOutput()` function does a keyword scan of the entire message content to guess output type. The new per-section save will receive the type from the detection function directly — more accurate and simpler. The old whole-message `saveOutput()` should remain untouched for backward compatibility (in case it's called from anywhere else).

### Finding 7: Existing animation-message-flash and message-appear Classes Available
**Source:** `globals.css` lines 102-137

The `message-appear` class (`animation: fade-in 0.3s ease-out`) and `animate-message-flash` are already defined. Card entrance animation via framer-motion's `initial/animate` props is preferred for staggered multi-card reveal (can delay per card index). The `message-appear` class can also work if framer-motion overhead is a concern for simple cards.

## Open Questions

1. **What does the AI output format actually look like for each type?**
   - What we know: The system prompt (not inspected — not in scope) instructs the AI to use `###` headings and field labels like `Subject:`, `Title:`, `Body:`.
   - What's unclear: Whether the AI consistently uses `**Subject:**` (bold) vs plain `Subject:`, and whether body text follows on the same line or next line.
   - Recommendation: Build the field parser defensively — handle both bold and plain, both same-line and next-line. Test with real AI outputs during implementation.

2. **Should the whole-message Save/Copy buttons be hidden when OutputCardGroup renders?**
   - What we know: Currently, Save and Copy are always shown below assistant messages in page.tsx.
   - What's unclear: If per-card Save/Copy exist, the outer buttons duplicate functionality.
   - Recommendation: When `isMultiSection` and not streaming, hide the outer Save/Copy buttons (they're for whole-message saves). Each card has its own. This is a page.tsx conditional to add.

3. **What happens to the adherence ScoreCard when OutputCardGroup renders?**
   - What we know: ScoreCard is rendered separately below each assistant message, after the message bubble, gated on `adherenceScores[message.id]`. It is independent of the message rendering.
   - What's unclear: Does the ScoreCard still make sense when the "message" is now N output cards?
   - Recommendation: Leave ScoreCard exactly as-is. It scores the entire response content which is correct — adherence scoring is holistic, not per-output-type. ScoreCard position (below the card group) is unchanged.
